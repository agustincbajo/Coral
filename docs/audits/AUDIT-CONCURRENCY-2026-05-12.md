# Audit: Concurrency — Coral v0.34.1
Date: 2026-05-12
Auditor: claude (sonnet)
Scope: workspace crates (`coral-cli`, `coral-core`, `coral-mcp`, `coral-runner`, `coral-ui`, `coral-session`, `coral-test`, `coral-env`, `coral-lint`, `coral-stats`). Focus: bootstrap LLM parallelism, `tiny_http` thread model (MCP + UI), multi-process `.wiki/` access, `fs4` advisory-lock semantics across Windows/Linux, `rayon` leverage, `--max-cost`/`--resume` race window, SessionStart concurrent invocations, `atomic_write_string` callsites, signal-driven shutdown plumbing, `coral monitor up` JSONL appends. Excluded: SPA-internal JS, real cross-OS contention runs, live `cargo audit` snapshot.

## Executive summary

Coral's concurrency story has two halves. **Single-process correctness is strong**: every persistent write (`.wiki/index.md`, `coral.lock`, `.claude/settings.json`, `.bootstrap-state.json`) goes through `atomic_write_string` (tmp+rename) wrapped by `with_exclusive_lock` (`fs4` advisory flock on a sibling `.lock` sentinel). The v0.15 multi-writer guarantee is pinned by `cross_process_lock_serializes_n_subprocess_increments`. **Multi-process orchestration around long-lived servers is weaker**: `coral ui serve` is single-threaded by construction (every request runs inline on the recv loop), the MCP HTTP transport spawns one `std::thread` per request but its 32-slot cap has no per-IP rate limit, and the MCP shutdown path calls `std::process::exit(0)` after 150 ms — shorter than a real `tools/call` round-trip, so in-flight responses can be truncated mid-write.

Three remediation priorities: (1) make `coral ui serve` request handling concurrent (small thread pool, NOT tokio); (2) widen MCP shutdown grace from 150 ms to ≥2 s OR drain an in-flight counter; (3) parallelize bootstrap per-page LLM calls with a bounded rayon pool, after fixing the non-atomic `cost_spent_usd` accumulator.

| CON-02 | Critical | MCP shutdown watchdog calls `std::process::exit(0)` after 150 ms regardless of in-flight handlers | `crates/coral-cli/src/commands/mcp.rs:330-339` | SIGINT flips the AtomicBool; watchdog sleeps 150 ms then exits. | CON-07 | High | SessionStart hook + `coral self-check --quick` race on `.coral/cache/self-check-quick.json` | `.claude-plugin/scripts/on-session-start.sh` → `crates/coral-cli/src/commands/self_check.rs` | Two Claude Code sessions open simultaneously on the same repo run the hook in parallel. ## Methodology

Static review only — read-only. Tools:

- `Glob` + `Grep` for `Arc<Mutex<…>>`, `Arc<AtomicBool>`, `signal_hook`, `fs4::`, `try_lock_exclusive`, `atomic_write_string`, `with_exclusive_lock`, `rayon::`, `par_iter`, `std::thread::spawn`, `incoming_requests`, `recv_timeout`, `O_APPEND`, `tiny_http::Server`.
- Traced two critical paths end-to-end: (a) `coral bootstrap --apply --max-cost=$0.50 --resume` against an existing checkpoint, (b) `coral mcp serve --transport http --bind 0.0.0.0` under simulated 50-concurrent-POST load.
- Diffed `coral ui serve` vs `coral mcp serve` to identify the asymmetric thread models (single-threaded inline vs thread-per-request). UI server's inline model is the most surprising finding.
- Confirmed `cross_process_lock.rs` passes the multi-process flock contract — `fs4` works at the OS-process boundary as advertised. Did NOT re-run the BACKLOG #8 `RUST_TEST_THREADS=2` adversarial repro; that path is documented as in-process-only by design.
- Cross-checked the prior multi-agent v0.30 audit `audit/raw/concurrency.md`; findings B1, B5, B6, B7, 010, M3 verified-still-applied.
- No load tests executed. This is paper review.

## Scope NOT audited

- **SPA-internal JS** (`crates/coral-ui/assets/dist/`) — Vite bundle, browser event loop only.
- **Live LLM provider rate-limit behavior** — Tier numbers in CON-04 are from public docs; not exercised against live API.
- **macOS `MoveFileEx` rename semantics** under heavy contention; `EXDEV` cross-fs path; APFS specifics.
- **Tokio migration cost-benefit beyond a one-line recommendation** (see Top-3). A real decision needs benchmark numbers.
- **Linux ETXTBSY repro** under `RUST_TEST_THREADS=2` — BACKLOG #8 closed; in-process `Mutex` mitigation documented.
- **`docker compose` / Podman backend races** (`coral env up`, `coral monitor up`) — external runtime, not Coral's contract.
- **CI runner parallelism** — `ci.yml` sets `RUST_TEST_THREADS=1` for deterministic suites; intentional per BACKLOG #8.
- **`coral test mutants` parallel execution** — wrapper untested under load (TEST-06).

## Top-3 next actions

1. **CON-01 + CON-06 in one PR.** `coral ui serve` and `coral mcp serve` both need a 250 ms `recv_timeout` discipline + per-request thread spawn with a static 8–16 slot semaphore. The UI server's single-threaded model is the more user-visible bug (browser-perceived freeze while LLM streams); fix shape is identical. ~200 LoC + one integration test per crate. **Do NOT switch to tokio** — workspace has explicitly chosen blocking I/O surface (Cargo.toml + ARCHITECTURE.md); the bug is solvable with `std::thread` + `Semaphore`-shaped permits.
2. **CON-02 (shutdown grace).** Replace the 150 ms sleep with an in-flight counter drain loop bounded by `CORAL_MCP_SHUTDOWN_GRACE_MS` (default 5000). `active: Arc<AtomicUsize>` is already exposed; the watchdog needs an `Arc` clone passed through. Same one-PR scope as CON-01.
3. **CON-09 (backup-inside-lock).** Move the `std::fs::copy` for the `.claude/settings.json.coral-backup-<ts>` write INSIDE the `with_exclusive_lock` closure. ~5 LoC. Closes a TOCTOU window any user running `coral self-register-marketplace` twice in parallel can trip — with zero detection because both invocations exit 0.

**Should `tiny_http` switch to tokio/hyper? No, not in v0.34.x.** Blocking-I/O posture is intentional (MSRV 1.85, single-binary distribution, no async-runtime supply chain). CON-01 + CON-02 are solvable with `std::thread::spawn` + `Semaphore` in <300 LoC. A tokio migration adds `tokio`, `hyper`, `bytes`, `pin-project-lite` — that's a v1.0 decision, not a v0.34 patch.

## Appendix: Medium + Low findings

- **CON-M01 (Medium):** `coral-core::cache::WalkCache::save` (`cache.rs:75-85`) uses `atomic_write_string` without `with_exclusive_lock`. Cross-process invocations (e.g. `coral lint` + `coral stats` in parallel) can torn-rename. Not data-corrupting (cache is regeneratable) but the "last writer wins" outcome is invisible.
- **CON-M02 (Medium):** `coral-core::log::WikiLog::append_atomic` uses `O_APPEND` — same Windows risk as CON-03 on `.wiki/log.md`. Lower probability (only `ingest`/`bootstrap`/`init` write the log), but two parallel `coral ingest --apply` invocations against the same `.wiki/` will interleave log lines on Windows.
- **CON-M03 (Medium):** `Mcp-Session-Id` minting (`http_sse.rs:824-840`) is not cryptographically random; cross-referenced as SEC-07 (High in security audit). Concurrency angle: two MCP HTTP servers on the same host start `AtomicU64` counters at 0 simultaneously, so cross-process session IDs can collide. Fix: `getrandom::fill`.
- **CON-M04 (Medium):** `coral env import --write` / `coral env devcontainer emit --write` (`env.rs:239, 278`) use `atomic_write_string` without `with_exclusive_lock`. Parallel invocations against the same destination torn-rename silently.
- **CON-M05 (Medium):** `coral-stats` not parallelized (per CON-10). Lifting would inflate the count past cap; logged here for v0.35 ratchet plan.
- **CON-L01 (Low):** `TMP_COUNTER` `AtomicU64` with `Relaxed` ordering. Fine for filename uniqueness (collisions statistically impossible per-process). Documented for clarity.
- **CON-L02 (Low):** `BootstrapLock::Drop` releases the flock best-effort and leaves the lockfile in place; intentional per `state.rs:340-345`. No issue.
- **CON-L03 (Low):** `test_script_lock` is per-test-binary (cargo runs `lib.rs` separately per binary). Cross-binary ETXTBSY theoretically possible. BACKLOG #8 closed; paper-risk classification holds.
- **CON-L04 (Low):** `coral-ui/src/server.rs:89-94` spawns "open browser" thread with no join handle. Leaks on Ctrl-C; Windows can leave a stray `start "" url` process. Cosmetic.
- **CON-L05 (Low):** `reap_expired_sessions` runs on every request (`http_sse.rs:845-849`). O(N) per request; negligible at <100 sessions but worth a note for the v0.35 multi-tenant story.
- **CON-L06 (Low):** `handle_get_sse` parks on `Condvar::wait_timeout` with the buffer mutex held during wait. Correct per std lib semantics (wait releases + reacquires), but subtle enough that a refactor could break it.
- **CON-L07 (Low):** `coral-runner::http::HttpRunner` SSE streaming has no soft-deadline; a misbehaving backend holds the per-request worker thread. The 32-slot cap eventually triggers 503s on MCP, but UI has no cap (CON-01). Lift to High only after CON-01 lands.
- **CON-L08 (Low):** `append_run` fsyncs every write (`monitor/run.rs:125`). Correct but ~5 ms on Windows NTFS / ~1 ms on Linux ext4. Invisible at 10s intervals; dominant cost at the v0.35 sub-second intervals.
- **CON-L09 (Low):** No deadlock potential observed; lock-acquisition graph has no nested `with_exclusive_lock` callsites. Worth pinning with a deadlock property test in `coral_core::tests::concurrency` for v0.35.
- **CON-L10 (Low):** `coral self-upgrade` has no flock around in-place binary replace. Two concurrent invocations both download+verify+rename; idempotent in practice but not formally safe. Out of scope (single-user tool).
